# Grill-me Session Record: ProductFlow SSO Config Hardening

This document captures the exact Q&A flow from the 2026-05-23 design session.
Each entry preserves the question, the candidate options actually presented,
the recommendation given at the time, and the final user pick. The aim is
two-fold:

1. Reviewers challenging a decision can read the original framing rather than
   reverse-engineering it from the PRD.
2. Future grill-me sessions on related work can reuse the candidate sets that
   proved decisive.

## Q1 - Scope tier

**Question**: What is the coverage scope of this redesign?

| Option | Content | Workload | Risk |
|--------|---------|----------|------|
| A small | shared_secret bug + zh.json + TTL helper text | 1d | near zero |
| B mid (recommended) | A + enable toggle + callback URL + test connection + 5-language i18n + secret strength + atomic save | 3-4d | introduces `productflow_sso.enabled` field + batch endpoint, requires migration |
| C large | B + status card with active sessions + force-logout + audit-log link + danger zone + import/export + model multi-select | 1-2w | ProductFlow co-changes (health endpoint, force-logout API) |

**Recommendation**: B. Reasons: P0 must ship; B contains the operational
primitives admins use daily; C requires cross-service endpoints that don't
exist yet and belongs in a separate wave.

**User pick**: B

## Q2 - SSO enable/disable mechanism

**Question**: How is the toggle implemented?

| Option | Field | Disable semantics | Backend changes |
|--------|-------|-------------------|-----------------|
| A independent enabled field | new `productflow_sso.enabled` (bool, default true) | config preserved; only refuses SSO when false | option row + validateForStart guard |
| B reuse base_url empty | no new field; base_url empty == disabled | must clear/restore base_url | none |
| C double check | enabled + base_url both must be non-empty | either empty -> disabled | same as A, but UI may say "enabled" while not working |

**Recommendation**: A. Real operational need to preserve config during
maintenance windows; B forces hostile workflows; C creates UI/state mismatch.

**User pick**: A

## Q3 - Status code when disabled

**Question**: What does `/api/productflow/sso/start` return when toggle is off?

| Option | Code | Semantics | ProductFlow mapping |
|--------|------|-----------|---------------------|
| A | 503 | Service Unavailable | reuses existing `error_sso_unavailable` variant |
| B | 404 | Not Found | inaccurate; endpoint exists |
| C | 410 | Gone | accurate but no existing variant |
| D | 403 | Forbidden | conflicts with "user disabled" 403 |

**Recommendation**: A. Reuses ProductFlow's existing error variant; consistent
with the current base_url-missing failure path; admin-vs-end-user distinction
handled via response body `message` field.

**User pick**: A

## Q4 - Startup WARN when disabled

**Question**: Does the Redis-fallback startup WARN fire when enabled=false?

| Option | Behavior |
|--------|----------|
| A | enabled=false -> no WARN. Add INFO line. |
| B | enabled=false -> still WARN ("future-proof") |

**Recommendation**: A. A disabled SSO cannot exercise the Redis fallback path,
so the warning is purely noise; INFO line lets ops confirm intentional state.

**User pick**: A

## Q5 - shared_secret blank-keep bug fix

**Question**: How to fix the "leave blank to keep" UX that actually clears?

| Option | Where | Trade-off |
|--------|-------|-----------|
| A front-end filter | `onSubmit` removes empty `shared_secret` from changedKeys | loses "actively clear secret" capability |
| B backend reject | `/api/option/` PUT returns 400 on empty secret for that key | special-casing in a generic endpoint |
| C `__KEEP__` protocol | special sentinel value | schema change, affects other secret fields |
| D separate reset button | secret moves to a Danger Zone | UI restructure, out of scope |

**Recommendation**: A. Placeholder text is already correct; backend stays
generic; "actively clear" is equivalent to "disable SSO", which Q2's toggle
handles.

**User pick**: A

## Q6 - Save atomicity

**Question**: How to handle the N-field save?

| Option | Implementation | Workload | Atomicity |
|--------|----------------|----------|-----------|
| A backend batch endpoint | `PUT /api/option/batch`, GORM transaction | 4-6h | true |
| B `Promise.allSettled` front-end | parallel N requests, aggregate failures | 1-2h | partial failure leaves bad state |
| C sequential with early stop | first failure halts, show prefix saved | 1h | same risk as B + harder recovery |
| D status quo | sequential N requests | 0 | broken |

**Recommendation**: A. base_url + secret must commit together; partial failure
= production-down. Pattern is reusable for other settings sections.

**User pick**: A

## Q7 - Test connection probe design

**Question**: What endpoint and what work crosses services?

| Option | ProductFlow exposes | new-api calls | Validates | Workload |
|--------|---------------------|---------------|-----------|----------|
| A lightweight probe | `GET /api/health/sso` returning `{ok, version, supports_sso}` | server-side fetch via `POST /api/productflow/sso/test` | base_url reachable + DNS + TLS + ProductFlow up + SSO module present | PF ~30 LOC |
| B full ping with secret | `POST /api/auth/new-api/verify-ping` requiring secret | simulate ticket exchange | A + secret match | PF 50+ LOC, ticket plumbing complex |
| C HEAD base_url | none | new-api HEAD | network only | 0 |
| D not implemented | - | - | - | 0, violates B-tier promise |

**Recommendation**: A. Covers 90% of misconfigurations; secret drift is rare
and exposed on first real login; ProductFlow side is trivial; C is actively
misleading (SPA catch-all routes return 200).

**User pick**: A

## Q8 - Callback URL display source

**Question**: Where does the displayed callback URL come from?

| Option | Source | Workload | Failure mode |
|--------|--------|----------|--------------|
| A front-end hardcoded constant | `removeTrailingSlash(base_url) + '/auth/new-api/callback'` | 5 LOC | drifts if ProductFlow changes path |
| B from health endpoint | A endpoint also returns `callback_path` | A change + PF change | cross-service contract auto-sync |
| C admin types it | manual `callback_path` field | 0 backend | error-prone |

**Recommendation**: A. Callback path is a two-side protocol contract, not an
admin choice; "auto-sync via health" is a feature-flag illusion - changing the
path actually breaks SSO so spec must be updated regardless.

**User pick**: A

## Q9 - UI layout for toggle + status

**Question**: Where does the toggle and the operational status live?

| Option | Layout |
|--------|--------|

| B | toggle as first form field, status hints scattered per-field |
| C | separate tab for Status vs Configuration |

**Recommendation**: A. Toggle is the highest-weight control; status card

**User pick**: A

## Q10 - TTL field unit UI

| Option | Presentation | Workload |

## Q11 - i18n strategy

**Question**: How to handle the 24 missing strings across 6 locales?

| Option | Workload | fr/ja/ru/vi display | Risk |
|--------|----------|---------------------|------|
| A zh only + sync | 24 zh | tool may auto-pick zh as base -> Chinese for fr/ja/ru/vi | uncontrolled |
| B zh + en (recommended) | 24x2 | fr/ja/ru/vi fall back to English (industry norm) | controlled |
| C zh only no sync | 24 | sync tool reports missing | tool fails |
| D 6 languages all hand-translated | 24x6 | full localization | translator quality risk on fr/ja/ru/vi |

**Recommendation**: B. en is i18next's fallback authority; explicit en keys
make minority locales' fallback English (not key strings); deferred fr/ja/ru/vi
translation can land in a future wave with the key structure already ready.

**User pick**: B (typed as lowercase "b")

## Q15 - Test-connection error granularity + timeout

**Question**: How to classify failures and how long to wait?

| Classification | |
|----------------|-|
| A single failure + raw message | low translation cost, low diagnostic value |
| B 5+ variants (dns/tcp/tls/timeout/401/etc.) | high translation cost |
| C 3 buckets: network / application / other (recommended) | balanced |

| Timeout | |
|---------|-|
| a 3 seconds (recommended) | snappy; deters slow domains |
| b 5 seconds | sluggish |
| c configurable | field pollution |

**Recommendation**: C + a. 3 buckets is enough for ops triage; 3s timeout
gives ProductFlow's <100ms health probe a 30x buffer; configurable would just
add noise.

**User pick**: C + a

## Q16 - shared_secret strength meter

**Question**: How does strength checking work?

| Option | Real-time | Submit-time |
|--------|-----------|-------------|
| A double layer (recommended) | strength bar (weak/med/strong) | schema `.min(16) + [A-Za-z] + [0-9]` |
| B submit only | none | min length check |
| C real-time only | strength bar | no enforcement |
| D nothing | - | - |

**Recommendation**: A. Real-time teaches admins about good secrets while
they type; submit guard prevents weak-but-lazy. Both gates skip when field
empty (preserves Q5 blank-skip).

**User pick**: A

## Q19 - SSO start/verify DB write level

**Question**: Do high-frequency SSO events write to the logs table?

| Option | start success | start failure | verify success | verify failure | Compliance |
|--------|---------------|---------------|----------------|----------------|------------|
| A | file only | file | file | file | logrotate eventually forgets |
| B | DB | DB | DB | DB | logs table bloat |
| C | DB + file | DB + file | DB + file | DB + file | double IO |
| D (recommended) | DB | DB | file only | DB | balanced |

**Recommendation**: D. Start success ties identity to action; verify success
is redundant once start is captured; failures are always compliance-relevant.
Volume estimate ~500 rows/day for 100 active users, ~54MB/year - negligible.

**User pick**: D

## Q20 - Toggle commit timing

**Question**: When does the toggle take effect?

| Option | Toggle click behavior | Trade-off |
|--------|----------------------|-----------|
| A instant | direct API call on toggle | accidental clicks can down production SSO |
| B form-dirty (recommended) | toggle becomes a dirty field; commit on Save | atomic with other config; safe |
| C A + AlertDialog | + confirmation popup | dialog fatigue |
| D direction-asymmetric | off=instant, on=via form | inconsistent behavior |

**Recommendation**: B. Consistent with Q6 batch atomicity; "must press Save"
is implicit confirmation that prevents typos from going live.

**User pick**: B

## Q21 - Test-connection URL source

**Question**: When admin changed base_url but didn't save, what URL does
Test Connection probe?

| Option | URL | UX |
|--------|-----|-----|
| A draft value from form (recommended) | what's in the input box right now | "edit then test" workflow |
| B saved value from OptionMap | the persisted value | misleading: admin changed URL, sees green check, thinks new URL worked |
| C refuse to test | must save first | anti-pattern |
| D let admin choose | toggle "saved/draft" | option overload |

**Recommendation**: A. Test-connection is read-only; using draft is zero-risk;
chip label distinguishes `draft URL` vs saved.

**User pick**: A

## Q22 - Audit write timing

**Question**: Inside or outside the GORM transaction?

| Option | Write location | Consistency | Risk |
|--------|----------------|-------------|------|
| A inside transaction (recommended) | tx.Create(&log) within db.Transaction | success/failure stay atomic | log write failure rolls back business (compliance first) |
| B after commit | commit business first, then write audit | audit may fail leaving compliance gap |
| C outbox pattern | tx writes outbox row, worker drains | over-engineered for single-DB audit |
| D outermost only | controller writes one aggregate audit after batch | granularity too coarse |

**Recommendation**: A. Compliance over business; failure probability is
minuscule (single-row insert in same tx); D fails Q18 detail dialog needs.

**User pick**: A

## Q23 - base_url validation strictness

**Question**: How strict is the URL validation?

| Option | Submit-time | Probe-time |
|--------|-------------|------------|
| A URL parse only | basic check | full error from probe | no deploy-mode hints |
| B HTTPS warning + loopback warning + private IP warning (recommended) | parse + warnings (non-blocking) | + probe | full signal, no false denial |
| C strict reject of http and non-public | hard reject | + probe | breaks dev workflows |
| D HTTPS warning only | mild | + probe | misses loopback |

**Recommendation**: B. Production deploys to `image.aync.cc.cd` must satisfy
three rules (HTTPS + public domain + correct reverse proxy); B surfaces them
all as amber warnings without blocking dev/internal scenarios.

**User pick**: B

## Q24 - shared_secret modification confirmation

**Question**: Extra friction for changing secret?

| Option | When | Friction | Catches |
|--------|------|----------|---------|
| A no extra friction | - | none | nothing - secret-change goes silent |
| B AlertDialog on Save if secret dirty (recommended) | once per Save | low | "I changed token_name and accidentally typed in secret too" |
| C inline warning on field focus | continuous | nag | low pickup rate |
| D B + C | overengineered |

**Recommendation**: B. Secret change has unique blast radius (all sessions
401 immediately, no rollback); B catches accidental secret changes cleanly.

**User pick**: B

## Q25 - Health probe schema + rate limit

**Question**: What does the ProductFlow health endpoint return, and how is it
protected from scanning?

| Schema | |
|--------|-|
| a minimal | `{ok}` |
| b medium (recommended) | `{ok, version, supports_sso}` |
| c complete | `{ok, version, supports_sso, expected_callback_path, max_session_ttl, redis_enabled}` |

| Rate limit | |
|------------|-|
| i none | hackers can scan freely |
| ii 6/min per IP (recommended) | enough for manual + automated checks; deters scanning |
| iii IP allowlist + ii | maintenance overhead in multi-instance deploys |

**Recommendation**: b + ii. Medium gives diagnostic value without leaking
deployment internals; 6/min limit is plenty for admin workflows.

**User pick**: b (assumed ii by recommendation default)

## Synthesis

The 25 decisions converge into a coherent design:

- **Compliance-first defaults**: Q19 (start success to DB), Q22 (audit in tx),
  Q24 (secret confirmation), Q23 (loud warnings on dangerous URLs).
- **Operational ergonomics**: Q2/Q9/Q14/Q20 form a coherent toggle + status
  card + dirty-state pattern; Q10/Q21/Q15 make the day-to-day admin tasks
  pleasant.
- **Minimal cross-service contract**: Q7/Q25 keep ProductFlow's required
  change to one rate-limited endpoint.
- **Reusable patterns**: Q6 batch endpoint, Q18 generic manage-changes table,
  Q11 i18n sync flow benefit future settings sections.
